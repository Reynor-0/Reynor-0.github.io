在图像、视频、音频和日志采集系统中，生产者和消费者很少天然同速。Camera ISP Pipeline 输出帧，V4L2、显示器、编码器或算法模块消费帧；网络接收线程写入包，业务线程解析包；日志线程写入事件，落盘线程批量刷盘。这些问题本质上都是有限缓冲区上的速率匹配问题。

本文用视频管线作例子，但设计方法适用于大多数生产者-消费者系统。

## 基础模型

一个常见模型是有限缓冲区池加两个队列：

```text
空缓冲队列                  满缓冲队列
等待生产者填充              等待消费者读取

[B0][B1][B2][B3]            []
   ^                          ^
生产者取空缓冲              消费者取满缓冲
```

队列操作通常可以抽象成：

- 生产者从空队列取 buffer，写入数据后放入满队列。
- 消费者从满队列取 buffer，处理完成后归还空队列。
- 如果要取的队列为空，可以选择阻塞等待、非阻塞返回、超时返回或丢弃策略。

需要注意的是，`O_NONBLOCK` 只改变“队列为空时是否等待”的行为，并不会自动实现“覆盖最老帧”。覆盖、丢弃最新帧、丢弃最旧帧，都必须由驱动、框架或应用层显式定义。

## 场景一：生产者快，消费者慢

这是最常见也最容易出问题的场景。核心矛盾是：消费者处理不过来时，系统到底保留低延迟，还是保留完整性。

### 延迟优先：只关心最新数据

实时预览、视频会议、遥控图传、监控预览通常更关心“现在看到什么”，而不是“每一帧都看到”。这类系统应使用小缓冲、非阻塞或超时等待，并主动丢弃旧数据。

可选策略有两种：

- **消费者侧 drain 队列**：消费者每次被唤醒后，把满队列中旧帧取出并归还，只处理最后一帧。
- **中间队列 drop-oldest**：满队列达到上限时，丢弃最旧的未消费帧，再插入新帧。

```text
Producer: 30 fps
Consumer: 5 fps

满队列: [F1, F2, F3, F4]
消费者醒来后丢弃 F1~F3，只处理 F4
```

这种模式延迟低，但会丢帧。不要让生产者覆盖消费者正在读取的同一块内存，否则会产生数据竞争。正确做法是通过 buffer ownership 明确规定：谁持有 buffer，谁才可以读写。

### 完整性优先：一帧都不主动丢

医疗采集、科学实验、离线素材导入、审计日志等系统更关心完整性。此时应使用阻塞等待和更大的缓冲区。当空队列耗尽时，生产者阻塞，背压向上游传递。

```text
消费者变慢
  -> 满队列累积
  -> 空队列耗尽
  -> 生产者取不到空 buffer
  -> 生产者阻塞或降速
```

背压的价值是让系统以消费者可承受的速度运行，而不是无限占用内存。代价是延迟会不断升高；如果上游是真实硬件传感器，硬件侧未必能无限等待，最终仍可能发生硬件层面的跳帧或溢出。

## 场景二：消费者快，生产者慢

这个场景相对简单。消费者没有数据可读时应进入等待状态，而不是忙轮询。

```text
t = 0 ms   消费者等待满队列
t = 40 ms  生产者写入 F1，唤醒消费者
t = 41 ms  消费者处理完成，再次等待
```

阻塞 `read`、阻塞 `DQBUF`、`select`、`poll`、`epoll` 本质上都可以睡眠等待内核事件；区别在于接口形态和能否同时等待多个 fd。它们不是忙轮询，正常使用时等待期间 CPU 占用应接近 0。

## 场景三：长期速率不匹配

有限缓冲只能吸收短期抖动，不能解决长期吞吐差距。只要生产速率长期高于消费速率，缓冲迟早会满。

```text
溢出时间 = 缓冲容量 / (生产速率 - 消费速率)

例如:
16 帧 / (30 fps - 25 fps) = 3.2 秒
```

这时必须改变系统吞吐，而不是继续加 buffer。

| 解法 | 思路 | 代价 | 适用场景 |
| --- | --- | --- | --- |
| 加速消费者 | SIMD、多线程、GPU、异步流水线 | 开发复杂度上升 | 消费逻辑可优化 |
| 降低生产速率 | 降帧率、降分辨率、采样、跳帧 | 信息损失 | 允许降质量 |
| 并行消费者 | 多 worker 分担不同任务或不同帧 | 需要处理顺序和状态 | 帧间依赖弱 |
| 磁盘解耦 | 先写入持久化队列，再离线处理 | 延迟高、依赖 IO | 离线分析 |
| 优先级丢弃 | 按重要性丢弃低价值数据 | 结果不完整 | 编码、遥测、监控 |

## 并行处理的边界

并行消费者能提高吞吐，但前提是任务可以拆分。逐帧独立的图像增强、格式转换、缩放、部分 ISP 算法通常适合并行；依赖历史状态的算法，例如时域降噪、运动估计、码流参考帧处理，则需要额外设计状态同步和输出排序。

```text
Producer -> Dispatcher -> Worker 1 -> Reorder/Merge
                      -> Worker 2 -> Reorder/Merge
                      -> Worker 3 -> Reorder/Merge
```

如果输出必须保持顺序，需要引入 sequence number 和 reorder buffer。否则并行处理会把吞吐问题转化为乱序问题。

## 设计决策树

```text
是否必须处理每一份数据？
├─ 是
│  ├─ 消费者平均吞吐 >= 生产者？ 使用阻塞 + 有限缓冲吸收抖动
│  └─ 消费者平均吞吐 < 生产者？ 必须加速消费者或降低生产速率
│
└─ 否
   ├─ 更关心实时性？ 使用小缓冲 + drop-oldest / drain-to-latest
   └─ 更关心平滑性？ 使用中等缓冲 + 允许延迟增长
```

一句话总结：**缓冲区解决的是短期抖动，不解决长期吞吐不匹配；非阻塞解决的是等待语义，不自动解决丢弃策略；实时性和完整性在消费者长期变慢时不能同时保证。**

## 工程检查清单

设计这类系统时，可以按下面的问题逐项确认：

- buffer 的所有权是否清晰，是否存在读写同一块内存的风险？
- 队列满时丢弃最新、丢弃最旧、阻塞还是返回错误？
- 队列空时阻塞、超时、非阻塞返回还是复用上一帧？
- 平均吞吐是否匹配，还是只靠大 buffer 延迟爆炸？
- 是否需要记录 dropped frame、queue depth、latency 等指标？
- 如果并行处理，是否需要输出重排序？
- 如果使用 V4L2，应用层是否正确处理 `EAGAIN`、`poll` 事件和 buffer 归还？

这些问题比“申请几个 buffer”更重要。buffer 数量只是结果，真正决定系统行为的是延迟、完整性、背压和丢弃策略之间的取舍。
